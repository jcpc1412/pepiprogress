# 09 — Internationalization

## Languages (6)
EN, ES, PT, FR, DE, **RU**. All Latin/Cyrillic — **no RTL work**. Catalogs scaffolded together so every new string lands in all six at once.

## The hard rule
**No hardcoded English, anywhere.** Every user-facing string comes from a catalog key.
- Stack: `i18next` + `react-i18next` + `expo-localization` for device-locale detection (overridable in-app).
- **Lint enforcement:** ESLint rule (`i18next/no-literal-string`) fails CI on raw JSX/text literals. English cannot sneak in.
- **Missing-key CI check:** a key present in one locale but absent in another fails the build. All six stay in lockstep.
- Keys are **semantic** (`log.sleep.prompt`), not English-derived, so reworded English doesn't churn other locales.

## Three classes of text (they localize differently)
1. **UI chrome** (buttons, labels, errors) — catalog strings, all 6 at launch. Non-negotiable per the rule.
2. **Catalog/medical-adjacent content** (compound aliases, cautions, side-effect labels, any education) — LLM-translated like the rest for MVP (most dosing/education copy is deferred anyway, 05). **Exception:** legally load-bearing strings — medical disclaimer, consent text (photo/biometric/community opt-in), safety cautions — get native-speaker review even at MVP (≈a dozen strings; mistranslation here is legal exposure, not a UX nit).
3. **User-generated content** (free notes, chat input, community text) — *not* translated; stored as written. The app *chrome* around it is localized.

## Formatting (locale, independent of language)
- **Units** — metric/imperial is a separate user setting, not tied to language (a US user wants English + imperial; a German lifter may want German + metric).
- **Dates, numbers, weights** — via `Intl`, per locale.
- **Pluralization** — i18next plural rules; **RU has complex plural forms** (one/few/many) — must use plural keys, not string concatenation.

## AI & i18n (ties to 05/13)
- **Conversational logging (13)** parses *and* responds in all 6 languages — a user logs "почувствовал тошноту утром" and it just works. This is a genuine edge; most quick-log features are English-only.
- **Voice quick-log** relies on device dictation, which supports all 6 languages.
- AI-generated text (insights, parsed-entry confirmations) must be returned in the user's locale — prompt the model with the target language.
- Compound names: international chemical/brand names stay; per-language **aliases** live in the catalog (08) so "arimidex"/"анастрозол" both resolve.

## Workflow
- English keys authored first → **LLM-translated** (Claude) into the other 5, all at once.
- Legally load-bearing strings (above) flagged for native review.
- Side-effect taxonomy and compound types stay as **enums**; only their display labels translate.
- Consider a translation-management tool (Crowdin/Locize) once string volume grows; not needed at MVP scale.

## Decisions (locked)
- 6 languages, Latin/Cyrillic only, no RTL.
- No hardcoded strings, lint + missing-key CI enforced.
- Units decoupled from language.
- UGC not translated; AI parses/responds per locale.
- **Translation: LLM-only for MVP**, except native review for legal/consent/safety strings.
- **Brand voice: informal/friendly everywhere** — informal "you" (ES tú / DE du / RU ты / PT tu / FR tu, not vous) across all languages, matching the community wellness vibe. Brief the translation prompt to use informal register consistently.
